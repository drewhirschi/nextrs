# Write-If-Changed Generated Assets

## Problem

Build scripts that generate public assets can rewrite watched files even when
bytes are unchanged. Because `nextrs-dev` watches `public`, this can trigger
unnecessary rebuild/restart churn. Tailwind-generated `public/style.css` is one
concrete example.

## Proposed Direction

Promote write-if-changed behavior across examples, docs, and generated assets:

- Framework examples should write generated public assets only when content
  changes.
- Docs should recommend write-if-changed for build-script outputs.
- The dev watcher should ignore known generated outputs where practical, or at
  least avoid restart loops caused by unchanged writes.

## Implementation Notes

- `nextrs::bundle` already has content-aware mirroring behavior. Reuse that
  idea for any public asset generation examples.
- For app-owned build scripts, provide a small copyable helper in docs rather
  than a framework dependency.
- Be careful with ignoring `public/dist`: it is generated, but changes there may
  still need browser reloads.

## Validation

- Add docs examples showing write-if-changed.
- Confirm generated scaffold code does not rewrite unchanged files in its own
  build pipeline.
- Manual dev check: run a build twice and confirm the watcher does not restart
  solely because generated bytes were rewritten.
